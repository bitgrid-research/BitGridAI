## 2. Datenaufzeichnung

### Was in der HA-DB steht (seit 2026-05-21)

Die Home-Assistant-SQLite läuft auf dem Umbrel und hält **365 Tage** Rohdaten.

| Sensor | Entity-ID | Einheit | Zweck |
|---|---|---|---|
| PV-Leistung | `sensor.pv_power_w` | W | Erzeugung |
| Hausverbrauch | `sensor.house_load_w` | W | Gesamtlast ohne Miner |
| Netzbezug | `sensor.grid_import_w` | W | Autarkie-Signal |
| Netzeinspeisung | `sensor.grid_export_w` | W | Überschuss-Signal |
| Akku-SoC | `sensor.battery_soc_pct` | % | Speicherstand |

| Miner-Leistung | `sensor.miner_total_power_w` | W | Verbrauch |
| Heizstab | `sensor.ac_elwa_2_..._power1_solar` | W | Flexible Last |
| Systementscheid | `sensor.bg_decision_action` | START/STOP/NOOP | Was hat HA entschieden? |
| Entscheidungscode | `sensor.bg_decision_code` | z.B. `START_R1_SURPLUS` | Welche Regel? |
| R1 aktiv | `binary_sensor.r1_surplus_ok` | on/off | Überschuss-Regel |
| R2 SoC hart | `binary_sensor.r2_soc_hard` | on/off | Akku-Schutz (hart) |
| R2 SoC weich | `binary_sensor.r2_soc_soft` | on/off | Akku-Schutz (weich) |
| R3 Übertemp | `binary_sensor.r3_overtemp` | on/off | Sicherheits-Stop |
| R5 Deadband | `binary_sensor.r5_deadband_active` | on/off | Stabilitäts-Puffer |
| Miner läuft | `binary_sensor.miners_any_running` | on/off | Betriebsstatus |

Granularität: ~1–2 Minuten (SMA-Polling). Der Szenario-Builder mittelt auf 10-Minuten-Blöcke.

## 3. Szenario-Builder

### Was er macht

1. **Export** — liest einen Tag (oder Zeitraum) aus der HA-SQLite
2. **Klassifikation** — bestimmt automatisch den Tagestyp
3. **Annotation** — schreibt Metadaten in den CSV-Header
4. **Bibliothek** — pflegt `library.json` als durchsuchbaren Index

### Tages-Klassifikation

| Tag-Typ | Kriterien | Forschungsrelevanz |
|---|---|---|
| `sunny_high` | PV-Peak > 7 kW, Import < 1 kWh | Optimales Mining-Fenster |
| `sunny_low` | PV-Peak 3–7 kW | Normalbetrieb |
| `cloudy` | PV-Peak < 3 kW | Regeltest: wie reagiert R1? |
| `night_only` | PV-Energie < 0.5 kWh gesamt | Reiner Netzbetrieb |
| `stress_soc` | min. SoC < 20% | R2-Grenzfall |
| `stress_overtemp` | max. Miner-Temp > 85°C | R3-Grenzfall |
| `grid_heavy` | Netz-Import > 5 kWh | Autarkie-Versagen |

Ein Tag kann mehrere Tags haben (z.B. `sunny_high + stress_soc`).

### CSV-Format der Szenarien

```
# scenario_builder: 1.0
# date: 2026-05-15
# tags: sunny_high
# pv_peak_kw: 9.2
# pv_energy_kwh: 42.1
# min_soc_pct: 34.0
# grid_import_kwh: 0.3
# grid_export_kwh: 18.5
# max_miner_temp_c: 71.2
# blocks_total: 144
# blocks_filled: 141
# timestamp_offset_min, pv_power_w, house_load_w, grid_import_w, battery_soc_pct,
#   miner_temp_c, miner_heartbeat_age_sec, energy_price_ct_kwh, pv_forecast_kw,
#   grid_export_w, miner_power_w, heizstab_power_w
0,8200.00,620.00,0.00,82.50,65.10,5.0,,,450.00,0.00,0.00
10,8450.00,640.00,0.00,83.20,65.40,5.0,,,520.00,0.00,0.00
...
```

144 Blöcke = 24h in 10-Minuten-Schritten.

### Alle Befehle

```bash
# DB holen
scp umbrel@umbrel.local:~/umbrel/app-data/home-assistant/data/home-assistant_v2.db /tmp/ha.db

# Einzelner Tag
python -m src.sim.scenario_builder build --db /tmp/ha.db --date 2026-05-15

# Zeitraum
python -m src.sim.scenario_builder build --db /tmp/ha.db --start 2026-05-01 --end 2026-06-01

# Bibliothek anzeigen
python -m src.sim.scenario_builder list
python -m src.sim.scenario_builder list --tag sunny_high
python -m src.sim.scenario_builder list --tag stress_soc

# Batch-Replay (alle Tage eines Typs durch die Simulation)
python -m src.sim.scenario_builder replay --tag sunny_high
python -m src.sim.scenario_builder replay --tag cloudy
```

## 4. Simulation (Replay)

Der Replay führt ein Szenario-CSV deterministisch durch den Entscheidungskern und gibt für jeden Block aus, was das System entscheiden würde.

```bash
# Einzelner Tag
python -m src.sim.replay --scenario src/sim/scenarios/real_2026-05-15.csv
```

### Ausgabe

```json
[
  {"block_id": "2026-05-15T00:00:00", "action": "STOP",  "decision_code": "STOP_R2_SOC_LOW",   "reason": "SOC_BELOW_THRESHOLD"},
  {"block_id": "2026-05-15T06:10:00", "action": "START", "decision_code": "START_R1_SURPLUS",  "reason": "SURPLUS_OK"},
  {"block_id": "2026-05-15T06:20:00", "action": "NOOP",  "decision_code": "NOOP_R5_DEADBAND",  "reason": "DEADBAND_HOLD"}
]
```

### Entscheidungscodes

| Code | Regel | Bedeutung |
|---|---|---|
| `START_R1_SURPLUS` | R1 Profitabilität | Mining startet wegen Überschuss |
| `STOP_R2_SOC_LOW` | R2 Autarkie | Akku zu leer |
| `STOP_R2_GRID_IMPORT` | R2 Autarkie | Netzbezug zu hoch |
| `STOP_R3_OVERTEMP` | R3 Sicherheit | Miner zu heiß |
| `STOP_R3_HEARTBEAT` | R3 Sicherheit | Miner nicht erreichbar |
| `NOOP_R4_FORECAST_LOW` | R4 Prognose | PV-Forecast zu gering |
| `NOOP_R5_DEADBAND` | R5 Stabilität | Zu früh nach letzter Änderung |

### KPIs aus Batch-Replay berechnen

```bash
# KPI-Tabelle für alle sonnigen Tage
python -m src.sim.scenario_builder replay --tag sunny_high

# Mit angepasster R1-Schwelle (z.B. 2.0 kW statt 1.5 kW)
python -m src.sim.scenario_builder replay --tag sunny_high --surplus-min-kw 2.0

# Ergebnisse als CSV für Jupyter/R exportieren
python -m src.sim.scenario_builder replay --tag sunny_high --csv /tmp/kpi_sunny.csv
```

Ausgabe (Spalten):

```
Datum         Uptime    Fenster%   Verlust kWh     R5%   FensterBlk
--------------------------------------------------------------------
  2026-05-22     8.3h      87.5%         1.23    12.5%          48
  2026-05-23    10.0h      95.0%         0.41     8.3%          60
--------------------------------------------------------------------
  ∅ 2 Tage      9.2h      91.3%         0.82    10.4%
```

| Spalte | Bedeutung |
|---|---|
| Uptime | Stunden, in denen der Miner tatsächlich lief (START + gehaltene NOOP) |
| Fenster% | Mining-Fenster-Ausnutzung: Uptime-Blöcke im Überschuss-Fenster |
| Verlust kWh | Netzeinspeisung in Blöcken, in denen Miner AUS war |
| R5% | Anteil der Blöcke, in denen R5 eine Entscheidung blockiert hat |
| FensterBlk | Blöcke mit PV-Überschuss ≥ R1-Schwelle (1.5 kW default) |

## 5. Architektur

```
scripts/
  ha_export_scenario.py     ← direkter DB-Export (auch standalone nutzbar)

src/sim/
  scenario_builder.py       ← Klassifikation + Annotation + Bibliothek
  scenario_loader.py        ← lädt CSV → EnergyState-Objekte
  replay.py                 ← führt Szenario durch Entscheidungskern
  runner.py                 ← zeitgesteuert (Echtzeit-ähnlich)
  scenarios/
    real_YYYY-MM-DD.csv     ← exportierte Realdaten
    library.json            ← Index aller Szenarien
    sh1_stable_surplus.csv  ← synthetische Testszenarien
    sh2_variable_pv.csv
    sh3_soc_critical.csv
    sh4_safety_overtemp.csv

src/core/                   ← Entscheidungskern (deterministisch, kein ML)
  rules/r1_*.py … r5_*.py  ← Regelmodule
  rule_engine.py
  models.py                 ← EnergyState, DecisionEvent

src/data/
  event_store.py            ← Entscheidungsprotokoll (SQLite, append-only)
```

**Schichtentrennung:** `core/` hat keine Imports von `sim/`, `ui/` oder `adapters/`.

## 6. KPI-Framework

Geordnet nach Forschungsfrage. Für jede Metrik: woher die Daten kommen und ob sie bereits verfügbar sind.

### Frage 1 — Wie gut nutzen wir den PV-Überschuss für Mining?

Das ist die Kernfrage des Projekts.

| KPI | Definition | Datenquelle | Status |
|---|---|---|---|
| **Mining-Fenster-Ausnutzung** | Miner-Uptime / Stunden mit PV-Überschuss (R1=ok) | Replay-Ergebnis + HA-DB | berechenbar |
| **Verlorener Überschuss** | grid_export_kWh an Stunden wo Miner AUS war | HA-DB | berechenbar |
| **Verpasste Starts (R4/R5)** | Blöcke: R1+R2+R3 ok, aber NOOP wegen R4 oder R5 | Replay-Entscheidungscodes | berechenbar |
| **Mining-kWh/Tag** | Tagesenergie der Miner | `mining_log.csv` | ✓ vorhanden |
| **Miner-Uptime/Tag** | Laufzeit in Stunden | `mining_log.csv` | ✓ vorhanden |

**Wichtigster Engpass:** Fenster-Ausnutzung und verlorener Überschuss — das zeigt ob R1-Schwelle (`surplus_min_kw = 1.5 kW`, konfigurierbar per `--surplus-min-kw`) zu konservativ ist.

---

### Frage 2 — Wie autark sind wir?

| KPI | Definition | Datenquelle | Status |
|---|---|---|---|
| **Autarkiequote** | (PV_kWh − grid_export_kWh) / house_kWh | HA-DB Integrale | berechenbar |
| **Eigenverbrauchsquote** | (PV_kWh − grid_export_kWh) / PV_kWh | HA-DB Integrale | berechenbar |
| **Netzbezug/Tag** | grid_import_kWh | HA-DB | ✓ vorhanden |
| **SoC-Unterschreitungen** | Blöcke mit SoC < 20 % | HA-DB | berechenbar |

---

### Frage 3 — Wie stabil und sicher verhält sich das System?

| KPI | Definition | Datenquelle | Status |
|---|---|---|---|
| **Start/Stop-Frequenz (Flapping)** | Zustandswechsel pro Tag | `mining_log.csv` (start/stop) | ✓ vorhanden |
| **R5-Deadband-Hit-Rate** | Anteil NOOP_R5_DEADBAND / Gesamtblöcke | Replay | berechenbar |
| **Thermale Ereignisse** | Blöcke mit Miner-Temp > 80 °C | HA-DB | berechenbar |
| **Safety-Lockout-Tage** | Tage mit R3-Stop | HA-DB `bg_decision_code` | berechenbar |

|---|---|

**F3: HA-Entscheidungen vs. Replay-Entscheidungen abgleichen**
`sensor.bg_decision_action` (was HA wirklich getan hat) ist jetzt in der DB. Das direkte Gegenüberstellen mit dem Replay-Ergebnis ist Phase-2-Arbeit.
